PGMais Team Dashboard

Proxy inteligente e cache da API do Jira Cloud para monitoramento em tempo real

O PGMais Dashboard é uma aplicação full-stack que conecta diretamente ao Jira Cloud da PGMais para exibir, em um painel unificado, o estado atual de todas as demandas da equipe de tecnologia. Ele resolve o problema de fragmentação de informações: ao invés de cada membro navegar pelo Jira procurando dados dispersos, o dashboard centraliza KPIs, backlogs, entregas e métricas de performance numa interface otimizada para TV e desktop.

Dashboard em Tempo Real

KPIs, cards de desenvolvedores, backlog e issues em andamento com atualização automática a cada 5 minutos.

IA Integrada

Assistente com Claude Sonnet 4.6 que analisa dados do Jira, prioriza demandas e responde perguntas em contexto.

Gestão Avançada

Métricas históricas de entrega, SLA, cycle time e rastreamento de performance por dev, produto e tipo.

✓ Benefícios

Cache Inteligente em 2 Camadas

Memória + SQLite com TTLs diferenciados (5 min dashboard, 1h gestão, 24h snapshots). Reduz chamadas ao Jira em até 95%.

Priorização com IA

Cada pedido de prioridade é avaliado pelo Claude Sonnet 4.6 com score de 0-500. BUs de gestão recebem multiplicador 1.5x.

Auto-Registro por Domínio

Usuários @pgmais e @ciclo são registrados automaticamente no primeiro login, com sync automático de BU.

Múltiplas Visualizações

Grid de devs, Kanban, Timeline, Produto, Gestão e Priorização — cada visão otimizada para um caso de uso.

Deltas e Snapshots de KPI

Snapshots de 24h permitem mostrar variação (↑↓) em cada KPI comparando com o estado anterior.

Segurança em Camadas

Rate limiting (100 req/min), JWT com expiração de 8h, sanitização de input, headers de segurança (CSP, X-Frame-Options).

📖 Manual de Uso

1. Instalação Rápida (Windows)

# Clone o repositório

git clone <repo-url> && cd Dashboards-pgmais

# Backend

cd backend

python -m venv .venv

.venv\Scripts\activate

pip install -r requirements.txt

cp .env.example .env # edite com suas credenciais

uvicorn main:app --reload --port 8000

# Frontend (outro terminal)

cd frontend

npm install

npm run dev

2. Docker (Produção)

docker-compose up --build -d

O docker-compose sobe backend (porta 8000) e frontend automaticamente.

3. Login

  1. Acesse http://localhost:5173 no navegador.
  2. Digite seu e-mail corporativo (@pgmais.com.br ou @ciclo.com.br).
  3. Usuários regulares não precisam de senha. Admins devem informar a senha configurada.
  4. O sistema auto-registra novos usuários e sincroniza a BU automaticamente.

Como Adicionar Novas Funcionalidades

Novo Endpoint na API

  1. Defina o modelo Pydantic em models.py.
  2. Adicione a rota em main.py com decorador @app.get/post/put/delete.
  3. Use security.py para validação de input e rate limiting.
  4. Se precisar de cache, use as funções de cache.py com TTL apropriado.

Nova View no Frontend

  1. Crie o componente em frontend/src/components/.
  2. Registre a view no estado de navegação em App.jsx.
  3. Adicione o link de navegação no header (dentro da nav do App.jsx).
  4. As chamadas HTTP usam fetch() nativo com token Bearer.

Nova Consulta ao Jira

  1. Adicione o JQL em jira_client.py.
  2. Use Semaphore(6) para respeitar o rate limit do Jira.
  3. A paginação via nextPageToken já está abstraída — use o padrão existente.

Fluxo de Cache

1. Request chega: Middleware valida JWT → Rate limiter verifica IP

2. Cache hit (RAM): Retorna dados do cache em memória (sub-ms)

3. Cache hit (SQLite): Fallback para cache persistente se RAM expirou

4. Cache miss: Busca no Jira → Parseia → Salva em ambas camadas → Retorna

5. Invalidação: POST /api/refresh limpa todas as chaves dashboard_* e mgmt_*

Dashboard: TTL 5 min Gestão: TTL 1 hora KPI Snapshot: TTL 24 horas

Segurança

Rate Limiting

  • API geral: 100 requisições / 60s por IP
  • Login: 15 tentativas / 15 min por IP

Autenticação JWT

  • Algoritmo: HS256
  • Expiração: 8 horas
  • Domínios permitidos: @pgmais, @ciclo

Validação de Input

  • Jira key: regex ^[A-Z][A-Z0-9]{0,9}-\d{1,6}$
  • Strings: max length + remoção de chars de controle
  • Listas: máximo 50 itens por parâmetro

Headers de Segurança

  • Content-Security-Policy
  • X-Frame-Options: DENY
  • X-Content-Type-Options: nosniff
  • Sanitização recursiva de dados sensíveis

Escala de Boost (IA)

400-500: Produção parada, perda financeira imediata

250-399: Impacto significativo em cliente major

100-249: Impacto moderado, cliente insatisfeito

0-99: Baixa urgência, sem impacto real

BUs gestão: multiplicador 1.5x (max 750)

curl -X POST \
