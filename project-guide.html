1. Visão geral

Frontend: React + Vite Backend: FastAPI Integração: Jira Cloud IA: OpenRouter / Claude Cache em memória

O projeto é um dashboard operacional e gerencial. Ele centraliza dados do Jira Cloud e os transforma em painéis que ajudam a responder perguntas como: quem está mais sobrecarregado, quais chamados estão atrasados, qual produto está pior, quais contas são mais críticas e o que merece priorização imediata.

Para leigos

Pense nele como uma “central de monitoramento” do time. Em vez de abrir muitos chamados manualmente no Jira, a pessoa vê em uma tela o estado do trabalho, os gargalos e os riscos.

Para devs

O frontend consome APIs próprias do backend. O backend fala com Jira, aplica parsing, calcula métricas, usa cache e expõe respostas mais prontas para a interface.

Objetivo real

Reduzir tempo para decisão. O sistema tenta transformar dados brutos em leitura rápida para operação, gestão e priorização.

O projeto já suporta operação local com dados mock quando credenciais do Jira não existem, o que facilita demonstrações e desenvolvimento.

2. Arquitetura do sistema

Usuário ↓ Frontend React (Vite) ├─ App.jsx controla telas, estado, filtros, busca e refresh ├─ Components mostram KPIs, backlog, cards, IA, produto, gestão e admin └─ apiUrl.js decide para qual backend apontar ↓ Backend FastAPI ├─ main.py expõe endpoints e regras de orquestração ├─ jira_client.py consulta Jira e converte dados crus em objetos úteis ├─ models.py define o formato oficial dos dados ├─ cache.py evita consultas repetidas ao Jira └─ mock_data.py mantém o sistema funcional sem credenciais ↓ Jira Cloud / OpenRouter / Arquivos JSON locais

Frontend

Responsável por experiência visual, navegação entre visões, filtros, busca textual, notificações e organização dos dados na tela.

Backend

Responsável por segurança das credenciais, integração com Jira, cache, consolidação dos dados, lógica histórica e recursos de IA.

A principal separação de responsabilidades está correta: credenciais e chamadas externas ficam no backend; exibição e interação ficam no frontend.

3. Funcionalidades existentes

Dashboard

KPIs, cards por desenvolvedor, backlog lateral, filtros múltiplos, busca e visualização rápida do trabalho ativo.

Gestão

Visão histórica para liderança com entregas por período, SLA, comparações semanais e radar por pessoa.

IA Insights

Perguntas em linguagem natural com contexto do dashboard para interpretação e recomendações.

Visão Produto

Comparativos entre produtos, temas recorrentes, prioridades, matriz analítica e drill-down de chamados.

Kanban

Agrupa jiras por estágio/status e ajuda a entender fluxo de trabalho.

Admin / BUs

Cadastro e manutenção de BUs, membros e tipos de unidade para alimentar filtros e regras de negócio.

Priorização

Score para ordenar urgências com ranking de accounts, classificação de produção e pedidos de prioridade com IA.

Resumo Noturno

Modo alternativo de leitura resumida para acompanhamento mais passivo.

Atualização manual

Botão para invalidar cache e buscar novos dados imediatamente.

4. Regras de segurança e pontos sensíveis

Medidas já presentes

  • Credenciais do Jira e da IA ficam no backend, não no navegador.
  • REFRESH_SECRET protege o refresh forçado quando configurado.
  • CORS é limitado para localhost, 127.0.0.1 e tunnels permitidos.
  • O endpoint de debug exige secret e valida o formato do ticket.
  • Há fallback para dados mock em desenvolvimento.

Atenções importantes

  • Arquivos JSON locais funcionam bem para desenvolvimento, mas não escalam para concorrência alta.
  • OpenRouter depende de chave ativa; falhas externas devem sempre ter fallback elegante.
  • Se o CORS for ampliado em produção sem critério, aumenta superfície de risco.
  • O endpoint de refresh sem secret em produção seria um problema operacional.
Em produção, o mínimo recomendado é: secret obrigatório, origem restrita, logs revisados, armazenamento persistente apropriado e variáveis sensíveis fora do repositório.

5. Usabilidade e experiência do usuário

Acertos

  • Busca textual rápida por chave, resumo ou pessoa.
  • Filtros multi-seleção por BU, responsável, account, produto, tipo e status.
  • Estados claros de carregamento e erro.
  • Alternância de tema claro/escuro e modo resumo.
  • Separação por visões ajuda cada perfil a ir direto ao que importa.

Melhorias possíveis

  • Adicionar legenda dos termos para quem não conhece Jira.
  • Melhorar acessibilidade de contraste, foco visível e navegação por teclado em todos os componentes.
  • Salvar filtros no navegador para o usuário voltar ao último contexto.
  • Adicionar ajuda contextual explicando cada KPI em linguagem simples.

6. Boas práticas percebidas no código

Separação por camadas

O backend faz integração e regra; o frontend renderiza. Isso reduz exposição indevida e facilita manutenção.

Reuso de componentes

O frontend está quebrado em componentes especializados, o que ajuda evolução visual e lógica isolada.

Uso de cache

Evita bater no Jira a cada carregamento, reduz latência e melhora estabilidade.

Boas decisões técnicas

  • useMemo e useCallback no frontend para evitar recomputações desnecessárias.
  • Modelos no backend padronizam a estrutura devolvida para a UI.
  • Fallback para mock acelera onboarding e testes locais.
  • Funções auxiliares pequenas em muitos componentes melhoram legibilidade.

Pontos para vigiar

  • App.jsx concentra muita responsabilidade e pode crescer demais.
  • Algumas telas possuem bastante lógica local e podem pedir extração de hooks/utilitários.
  • Arquivos muito longos, como gestão e produto, tendem a ficar mais difíceis de manter com o tempo.

7. Fluxos principais do sistema

Fluxo do dashboard

  1. O navegador abre o frontend.
  2. App.jsx chama /api/dashboard.
  3. O backend usa cache ou busca no Jira.
  4. Os dados voltam consolidados para a tela.
  5. O usuário filtra, busca, muda de visão e atualiza se necessário.

Fluxo da IA

  1. A tela monta um contexto textual com dados do dashboard.
  2. O backend valida a pergunta e a chave de IA.
  3. O prompt é enviado ao OpenRouter.
  4. A resposta é higienizada e exibida para o usuário.
A IA é útil para interpretação, mas decisões críticas continuam devendo revisão humana, especialmente em priorização e impacto em produção.

9. Onde alterar cada tipo de necessidade

Se quiser mudar...

  • URL da API → frontend/src/apiUrl.js
  • Filtros, busca e abas principais → frontend/src/App.jsx
  • Cards e backlog do dashboard → componentes base do dashboard
  • Dados vindos do Jira → backend/jira_client.py
  • Formato da resposta da API → backend/models.py

...ou ajustar regra de negócio

  • Refresh manual e cache → backend/main.py + backend/cache.py
  • Ranking de accounts → endpoints admin + account_ranking.json
  • Pedidos de prioridade com IA → create_priority_request() e _evaluate_priority_with_ai()
  • Histórico gerencial → get_management() e build_management_data()
  • Experiência das abas específicas → componente da aba correspondente

10. Riscos, débitos e sugestões de evolução

Curto prazo

  • Quebrar App.jsx em hooks e módulos menores.
  • Adicionar guia visual com explicação dos KPIs.
  • Persistir filtros do usuário em localStorage.

Médio prazo

  • Migrar JSONs locais para banco ou storage mais robusto.
  • Criar testes automatizados para backend e para funções críticas de priorização.
  • Padronizar utilitários compartilhados entre componentes grandes.

Longo prazo

  • Autenticação e perfis de acesso.
  • Observabilidade mais forte com métricas, tracing e alertas.
  • Exportação de relatórios e trilha de auditoria de mudanças.
Resumo executivo: a base do projeto é boa, útil e com separação saudável entre backend e frontend. O principal cuidado daqui para frente é evitar concentração excessiva de lógica em arquivos muito grandes e fortalecer persistência, acessibilidade, testes e segurança operacional.
